What I Learned Migrating 4 Production Microservices to Spring Boot 3 & Elasticsearch 8

By Anshika Singh May 20, 2026 8 min read

Earlier this year at Kruzr, I was handed a task that looked straightforward on paper: migrate 4 production microservices from Spring Boot 2.x to 3.x and from Elasticsearch 6.x to 8.x. No big deal, right? Famous last words. What followed was one of the most educational few weeks of my career — a deep dive into breaking changes, deprecated APIs, and the kind of debugging that only production systems can produce.

This post is the guide I wish I'd had before I started.

"The most dangerous assumption in any migration is thinking it's mostly a version bump."

Why migrate at all?

Spring Boot 2.x reached end-of-life in November 2023, which means no more security patches. Elasticsearch 6.x was even further behind — released in 2017, it lacked critical features our team wanted to use and was increasingly painful to maintain. The migration wasn't optional; it was overdue.

Spring Boot 3 also brings real benefits: native GraalVM support, improved observability with Micrometer, better virtual thread support in Java 21, and a cleaner security model. The pain is real but so is the payoff.

The jakarta.* rename — the one that breaks everything

This is the first thing that will hit you, and it hits hard. Spring Boot 3 is built on Jakarta EE 9+, which means the entire javax.* namespace has been renamed to jakarta.*. Every import. Every annotation. Every reference.

What breaks

  • javax.persistence.* → jakarta.persistence.* (all JPA annotations)
  • javax.validation.* → jakarta.validation.* (Bean Validation)
  • javax.servlet.* → jakarta.servlet.* (filters, requests)
  • javax.annotation.* → jakarta.annotation.*
Java · Before (Spring Boot 2.x)
import javax.persistence.Entity;
import javax.persistence.Id;
import javax.validation.constraints.NotNull;
import javax.servlet.http.HttpServletRequest;
Java · After (Spring Boot 3.x)
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.validation.constraints.NotNull;
import jakarta.servlet.http.HttpServletRequest;
💡 Practical tip

Use your IDE's Find & Replace (Cmd+Shift+R in IntelliJ) across the whole project: replace javax. with jakarta. — but be careful with javax.sql and javax.crypto, those don't change. Review each replacement.

Spring Security 6 — WebSecurityConfigurerAdapter is gone

If your services use Spring Security, brace yourself. The WebSecurityConfigurerAdapter pattern — the standard way to configure security in Boot 2.x — has been completely removed. Spring Boot 3 uses a component-based model with SecurityFilterChain beans instead.

Java · Old pattern (removed in Boot 3)
// This no longer compiles in Spring Boot 3
@Configuration
public class SecurityConfig extends WebSecurityConfigurerAdapter {
  @Override
  protected void configure(HttpSecurity http) throws Exception {
    http.authorizeRequests().anyRequest().authenticated();
  }
}
Java · New pattern (Spring Boot 3 / Security 6)
@Configuration
@EnableWebSecurity
public class SecurityConfig {

  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) 
      throws Exception {
    http.authorizeHttpRequests(auth ->
      auth.anyRequest().authenticated()
    );
    return http.build();
  }
}

Elasticsearch 6 → 8 — The client is completely different

This was the bigger challenge. Elasticsearch 8 didn't just change APIs — it removed the entire High Level REST Client (HLRC) that most Spring Boot apps relied on. The replacement is the new Elasticsearch Java API Client, and it works completely differently.

What was removed

  • The RestHighLevelClient — gone entirely in 8.x
  • Old query builders like QueryBuilders.matchQuery()
  • The org.elasticsearch.client package structure

The new client setup

Java · New Elasticsearch Java Client (8.x)
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.json.jackson.JacksonJsonpMapper;
import co.elastic.clients.transport.ElasticsearchTransport;
import co.elastic.clients.transport.rest_client.RestClientTransport;

RestClient restClient = RestClient.builder(
  new HttpHost("localhost", 9200)
).build();

ElasticsearchTransport transport = new RestClientTransport(
  restClient, new JacksonJsonpMapper()
);

ElasticsearchClient client = new ElasticsearchClient(transport);

Query syntax changes

The query DSL changed from a builder-of-builders pattern to a more fluent lambda-based API:

Java · Old (HLRC) vs New (ES 8 Java Client)
// Old — ES 6.x with High Level REST Client
SearchRequest request = new SearchRequest("rides");
request.source().query(QueryBuilders.matchQuery("status", "completed"));

// New — ES 8 Java Client
SearchResponse<Ride> response = client.search(s -> s
  .index("rides")
  .query(q -> q
    .match(m -> m
      .field("status")
      .query("completed")
    )
  ), Ride.class);

Security enabled by default in ES 8

Elasticsearch 8 ships with security enabled out of the box — TLS and authentication are mandatory. In development this caught us off guard because our services would start up fine but fail to connect. You either need to configure HTTPS properly or explicitly disable security in elasticsearch.yml for local dev:

elasticsearch.yml · Dev only
xpack.security.enabled: false
xpack.security.http.ssl.enabled: false
⚠️ Warning

Never disable security in production. Set up proper TLS certificates and use the built-in Elasticsearch keystore for credentials in production environments.

Lessons that actually stuck

  • Read the migration guide before you touch any code. Both Spring and Elastic have official migration guides. I skimmed them at first — big mistake. Read them fully, then start.
  • Migrate one service at a time. Don't try to upgrade all 4 simultaneously. Pick the smallest, least-critical service, get it fully working, then use it as your template.
  • Your test suite will save you. Services with good integration tests surfaced breaking changes immediately. The ones with sparse tests needed more manual verification — far slower and riskier.
  • Data consistency gaps are sneaky. After the ES migration, some indexed documents had fields that didn't match the new mapping. Always re-index or validate your data as part of the migration plan.
  • Check your dependencies, not just your code. Third-party libraries that depend on javax.* or the old ES client will also break. Run mvn dependency:tree and look for conflicts.

"A migration is never just a version number change. It's a small archaeology project — you find out what assumptions your codebase was quietly making all along."

Final thoughts

The migration took longer than expected, but each service we migrated made the next one faster. The pattern recognition kicks in — you start to instantly spot a javax import or an old QueryBuilders reference and know exactly what to do.

If you're facing a similar migration, take it one service at a time, read the official guides carefully, and trust your tests. It's not as scary as it looks — it's just a lot of careful, methodical work.
