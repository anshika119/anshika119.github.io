Why One Database Is Never Enough — The Case for Polyglot Persistence

By Anshika Singh May 10, 2026 6 min read

When I started building QueryMesh — a personal project exploring multi-database architecture — I kept running into the same question: why do so many real-world systems use multiple databases? Isn't that unnecessarily complex? Can't one good database just handle everything?

The short answer is: it can handle everything, but not everything well. After building QueryMesh and seeing how different databases behave under different access patterns, the case for polyglot persistence became obvious. Here's what I learned.

"Every database is a set of trade-offs optimised for a specific access pattern. Using one database for everything means most of your data is stored sub-optimally."

What is polyglot persistence?

The term was coined by Martin Fowler and Pramod Sadalage in their book NoSQL Distilled (2012). The idea is simple: instead of forcing all your data into a single database, you choose the right database for each type of data and access pattern.

It's the database equivalent of using the right tool for the job. You wouldn't use a screwdriver to hammer a nail — so why use a relational database for full-text search, or a document store for financial transactions?

The four database types and what they're good at

PostgreSQL / MySQL
Relational
ACID transactions, complex joins, structured data with clear relationships. The right choice when data integrity is non-negotiable.
→ Trip records, user accounts, payments
MongoDB
Document Store
Flexible schema, hierarchical data, rapid iteration. Great when your data structure isn't fixed and you need to store nested objects.
→ Event logs, user-generated content, config
Redis
In-Memory Key-Value
Sub-millisecond reads, caching, session storage, pub/sub messaging. Fast because everything lives in RAM — not durable by default.
→ Real-time vehicle states, cached risk scores
Elasticsearch
Search Engine
Full-text search, complex aggregations, analytics over large datasets. Built for searching, not for transactions — eventual consistency.
→ Ride search, driver analytics, log queries

How I built QueryMesh

QueryMesh is a Spring Boot service that acts as a routing layer between client services and four different databases. The idea: client services don't need to know which database to talk to — they just describe what they need, and QueryMesh routes accordingly.

The routing logic is straightforward but powerful:

  • Structured trip records with ACID requirements → PostgreSQL
  • Unstructured event logs with variable schema → MongoDB
  • Real-time vehicle states that need sub-millisecond reads → Redis
  • Search queries over ride data → Elasticsearch
Java · QueryMesh routing logic (simplified)
public class QueryRouter {

  public Object route(DataRequest request) {
    return switch (request.getType()) {
      case TRANSACTIONAL  -> postgresRepo.save(request);
      case EVENT_LOG      -> mongoRepo.insert(request);
      case REALTIME_STATE -> redisTemplate.opsForValue()
                              .set(request.getKey(), request.getValue());
      case SEARCH         -> elasticClient.search(request);
      default -> throw new UnsupportedOperationException();
    };
  }
}

The real trade-offs you need to know

Polyglot persistence isn't free. Before adopting it, you need to honestly assess the costs:

Operational complexity

You're now running, monitoring, backing up, and scaling four different database systems. Each has its own failure modes, its own backup strategy, its own scaling story. This is a significant DevOps burden — only justified if the data access improvements are real.

Data consistency across systems

In a single relational database, transactions guarantee consistency. Across multiple databases, you lose that guarantee. If you write a trip record to PostgreSQL and then the Elasticsearch indexing fails, your search index is out of sync. You need to design for this — event sourcing, saga patterns, and idempotent operations become important.

Query complexity

Queries that join data across databases don't exist at the DB level — they have to be assembled in application code. In QueryMesh, this meant some responses required two separate database calls merged in the service layer. Acceptable for most use cases, but something to design around.

"The routing layer is only as good as the thought that went into deciding what goes where. Map your access patterns first, build the router second."

When should you use polyglot persistence?

Not every application needs it. A simple CRUD app with uniform data access patterns is fine on a single PostgreSQL database. Consider polyglot persistence when:

  • You have genuinely different data types with different access patterns
  • You're hitting performance limits on a single database for specific query types
  • You need full-text search that relational databases handle poorly
  • You have real-time data that needs sub-millisecond access
  • Your team has the operational maturity to manage multiple systems

The pattern is common in high-scale systems — Netflix, Airbnb, Uber all use multiple databases. But they got there because they needed to, not as a starting point.

The takeaway

Building QueryMesh gave me a hands-on understanding of how different databases actually behave — not just in theory but under real query patterns. The biggest lesson: don't choose your database based on what you're familiar with. Choose it based on what your data needs.

Understanding polyglot persistence makes you a better architect. It forces you to think clearly about data access patterns, trade-offs, and system boundaries — skills that apply whether you end up using one database or five.
